# External Insights Policy

本文件規範 `project-insight-synthesizer` 如何處理 SWOT、競品、產業資料與額外建議。

## 核心原則

### 1. 會議事實與外部洞察必須分層
- **會議事實層**：只來自逐字稿、會議記錄、使用者提供文件
- **外部洞察層**：來自網路搜尋、產業報告、競品網站、新聞、研究機構、官方文件

兩者不得混寫。

### 2. 任何額外洞察都必須附上資料連結
以下內容若不是會議原文直接提到，都必須附上 URL：
- SWOT
- 競品分析
- 市場規模 / 產業趨勢
- 產品優勢 / 劣勢推論
- 建議改善方向

## 推薦輸出做法

### A. 在主文中加註來源類型
範例：
- 會議事實：`來源：2026-04-15 逐字稿`
- 外部洞察：`外部 insights 來源：NVIDIA Isaac Sim — https://developer.nvidia.com/isaac/sim`

### B. 若外部資料不足，不要硬做 SWOT
若無法取得足夠可信資料：
- 寫「外部資料不足，暫不做完整 SWOT」
- 或只寫局部觀察，但仍附來源

### C. 明確標示這是分析，不是會議共識
例如：
- `以下為外部市場洞察，非本次會議直接結論。`
- `以下 SWOT 為根據外部資料與會議內容交叉整理之分析。`

## 可接受的外部來源類型
- 官方網站 / 官方文件
- 產品頁 / 技術文件 / 開發者文件
- 正規媒體報導
- 研究機構 / 顧問公司報告
- 開源專案官方 repo / 官方 docs

## 盡量避免
- 無來源的二手摘要
- 內容農場
- 只有單一論壇留言、沒有官方佐證的說法
- 沒有日期與出處的市場數字

## 對使用者的回報要求
若加入 SWOT / 外部洞察，回報時要告知：
1. 哪些段落是會議原始內容
2. 哪些段落是外部 research / insights
3. 外部資料是否足夠完整
4. 是否存在高不確定性判斷
